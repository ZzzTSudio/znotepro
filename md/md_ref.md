# MD规范文档

前言部分：此部分用于总结当前 Markdown 文件做了什么功能、解决什么问题、面向什么读者，以及文档的主要内容范围。前言应使用自然段描述，不建议写得过长，通常 1 到 3 段即可。

示例：

本文档用于说明传统语音降噪算法的基本原理、常见算法分类、工程选型方法和实验验证流程。文档面向需要学习、实现或比较语音降噪算法的开发者、研究人员和工程实践者。

## 1. 一级标题规范

一级标题用于表示文档的主要章节。除文档最顶部的 `# 文档标题` 外，正文中的主要章节统一使用二级标题：

```md
## 1. 标题1
```

要求如下：

- 标题必须带数字编号。
- 编号从 `1` 开始递增。
- 标题名称应简洁明确。
- 同级标题格式必须保持一致。
- 标题后应先写正文内容，不建议标题下面直接连续堆叠多个子标题。

示例：

```md
## 1. 前置知识

填写本章节的说明内容。
```

## 2. 二级标题规范

二级标题用于表示某个主要章节下的细分内容。统一使用三级标题：

```md
### 1.1 标题1
```

要求如下：

- 子标题编号应跟随所属章节。
- 例如第 2 章下的子标题应写作 `2.1`、`2.2`、`2.3`。
- 不建议跳级使用标题，例如从 `##` 直接跳到 `####`。
- 子标题内容应服务于上一级标题。

示例：

```md
## 2. 算法分类

填写本章节总述内容。

### 2.1 谱减法

填写谱减法相关内容。

### 2.2 维纳滤波

填写维纳滤波相关内容。
```

## 3. 正文内容规范

正文内容用于解释概念、描述流程、说明背景或总结结论。正文应使用完整句子表达，避免只写零散关键词。

要求如下：

- 每个章节开头应有简短说明。
- 每段只表达一个核心意思。
- 段落之间保留一个空行。
- 重要概念可使用加粗标记。
- 文件名、函数名、参数名、命令、变量名应使用行内代码格式。

示例：

```md
语音信号具有短时平稳特性，因此在进行频谱分析前，通常需要先进行分帧处理。

其中，`frame_len` 表示帧长，`frame_shift` 表示帧移。
```

## 4. 列表规范

列表用于表达并列内容、步骤、规则或注意事项。

无序列表适合表达并列规则：

```md
- 内容1
- 内容2
- 内容3
```

有序列表适合表达流程步骤：

```md
1. 第一步。
2. 第二步。
3. 第三步。
```

要求如下：

- 同一组列表应使用同一种列表类型。
- 列表项应保持语法结构一致。
- 列表前应有一句引导说明。
- 列表后如需补充说明，应另起自然段。

## 5. 表格规范

表格用于展示结构化信息，例如参数说明、算法对比、字段定义、版本记录等。

标准写法如下：

```md
| 字段 | 说明 | 示例 |
| --- | --- | --- |
| 字段1 | 填写说明 | 填写示例 |
| 字段2 | 填写说明 | 填写示例 |
```

要求如下：

- 表格前应说明表格用途。
- 表头命名应简短清晰。
- 每一行字段含义应完整。
- 表格不宜过宽，列数建议控制在 3 到 6 列。
- 如果内容过长，应改为列表或拆分为多个表格。

示例：

| 参数 | 含义 | 推荐值 |
| --- | --- | --- |
| `frame_len` | 帧长 | 25ms |
| `frame_shift` | 帧移 | 10ms |
| `sample_rate` | 采样率 | 16kHz |

## 6. 公式规范

公式用于表达数学关系、算法模型或计算规则。

独立公式应使用块级公式：

```md
$$
公式
$$
```

示例：

$$
y(t) = s(t) + n(t)
$$

正文中的短公式可使用行内公式，例如 `$x + y = z$`。

要求如下：

- 重要公式应单独成段。
- 公式前应说明公式含义。
- 公式后应解释变量含义。
- 不应只放公式而不解释。

变量说明示例：

| 符号 | 含义 |
| --- | --- |
| $y(t)$ | 带噪语音 |
| $s(t)$ | 纯净语音 |
| $n(t)$ | 噪声 |

## 7. 代码块规范

代码块用于展示程序、命令、配置或 Markdown 示例。

标准写法如下：

````md
```python
def example():
    return "hello"
```
````

要求如下：

- 代码块必须标注语言类型。
- 代码前应说明代码用途。
- 代码后应补充关键逻辑说明。
- 不建议放过长代码，长代码应拆分为多个小段。

常见语言标记：

| 类型 | 标记 |
| --- | --- |
| Python | `python` |
| JavaScript | `javascript` |
| Shell | `bash` |
| PowerShell | `powershell` |
| JSON | `json` |
| Markdown | `md` |

## 8. 图片规范

图片用于辅助说明结构、流程、效果或对比结果。

标准写法如下：

```md
![图片说明](图片路径)
```

要求如下：

- 图片说明必须能概括图片内容。
- 图片路径应使用相对路径或明确的文件路径。
- 图片前应说明为什么需要该图。
- 图片后应解释图中关键信息。

示例：

```md
下图展示了 STFT 的基本处理流程。

![STFT处理流程](stft_process.png)

从图中可以看到，语音信号需要依次经过分帧、加窗、FFT 和频谱处理。
```

## 9. 引用与注释规范

引用用于保留原文、论文、标准或外部资料中的关键内容。

标准写法如下：

```md
> 引用内容
```

要求如下：

- 引用内容应尽量简短。
- 引用后应说明其含义或用途。
- 引用外部资料时应标明来源。
- 不应大段复制原文替代自己的总结。

脚注可用于补充来源或解释：

```md
这是正文内容。[^1]

[^1]: 这里填写补充说明或来源。
```

## 10. 章节结尾规范

每个重要章节可以在结尾增加简短总结，用于帮助读者理解本章重点。

推荐写法：

```md
本节总结：填写本节最重要的结论。
```

要求如下：

- 总结应简短。
- 总结应提炼结论，不重复全文。
- 技术文档中建议在复杂章节后添加总结。

## 11. 文件整体结构规范

一个完整 Markdown 文档建议采用以下结构：

```md
# 文档标题

前言部分：此部分总结目前文件做了什么功能。

## 1. 标题1

填写内容。

## 2. 标题2

填写内容。

### 2.1 子标题

填写内容。

## 3. 标题3

填写内容。

$$
公式
$$

## 4. 总结

填写总结内容。
```

## 12. 通用写作要求

所有 Markdown 文档应遵循以下通用要求：

- 标题层级清晰。
- 章节编号连续。
- 正文表达完整。
- 表格格式统一。
- 公式前后有解释。
- 代码块标明语言。
- 图片有说明文字。
- 同一文档内标点、术语和格式保持一致。
- 不同类型文档可扩展章节，但基础结构应保持统一。

## 13. 推荐模板

新建 Markdown 文档时，可直接使用以下模板：

```md
# 文档标题

前言部分：此部分总结目前文件做了什么功能。

## 1. 背景说明

填写本文档的背景、用途和适用范围。

## 2. 核心内容

填写文档的主要内容。

### 2.1 子内容1

填写子内容。

### 2.2 子内容2

填写子内容。

## 3. 关键规则

填写规则、流程、参数或注意事项。

| 字段 | 说明 |
| --- | --- |
| 字段1 | 填写说明 |
| 字段2 | 填写说明 |

## 4. 公式或示例

填写公式说明。

$$
公式
$$

填写公式解释。

## 5. 总结

填写本文档的最终结论。
```

## 14. 总结

本文档定义了一套通用 Markdown 写作规范，适用于技术文档、算法说明、项目方案、接口文档、操作手册、学习笔记和总结报告等多种场景。所有文档都应以清晰标题、完整正文、统一表格、规范公式和明确总结为基础，保证不同文件之间的阅读体验和结构风格一致。
